1. An adversary can gain unauthorized access to Azure Data Lake Store due to weak access control restrictions  [State: Not Started]  [Priority: High] 

Category:Elevation of Privileges
Description:An adversary can gain unauthorized access to Azure Data Lake Store due to weak access control restrictions
Justification:Managed Identity will be used on top of service principal to connect securely Azure Stream Analytics service to Azure Data Lake Store.
Possible Mitigation(s):Grant limited access to objects in Azure Storage using SAS or SAP. It is recommended to scope SAS and SAP to permit only the necessary permissions over a short period of time. Refer: https://aka.ms/tmt-th17a and https://aka.ms/tmt-th17b
SDL Phase:Implementation

2. An adversary may gain unauthorized access to Azure Data Lake Store account in a subscription  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation of Privileges
Description:An adversary may gain unauthorized access to Azure Data Lake Store account in a subscription
Justification:Configured Managed Identity (system-assigned identity to authenticate to other services) to allow Azure Stream Analytics to write to storage.
Possible Mitigation(s):Assign the appropriate Role-Based Access Control (RBAC) role to users, groups and applications at the right scope for the Azure Storage instance. Refer: https://aka.ms/tmt-th67
SDL Phase:Implementation

3. An adversary can abuse poorly managed Azure Data Lake Store account access keys  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation of Privileges
Description:An adversary can abuse poorly managed Azure Data Lake Store account access keys and gain unauthorized access to storage.
Justification:No access key used with Data Lake Store Gen 1; access is done through managed identity maintained in Azure Active Directory
Possible Mitigation(s):Ensure secure management and storage of Azure storage access keys. It is recommended to rotate storage access keys regularly, in accordance with organizational policies. Refer: https://aka.ms/tmt-th63
SDL Phase:Implementation

4. An adversary can deny actions on Azure Data Lake Store due to lack of auditing   [State: Mitigation Implemented]  [Priority: Medium] 

Category:Repudiation
Description:Proper logging of all security events and user actions builds traceability in a system and denies any possible repudiation issues. In the absence of proper auditing and logging controls, it would become impossible to implement any accountability in a system.
Justification:After this proof of value, if successfull, diagnostic logging will be enabled in production for our Data Lake Storage Gen1 account
Possible Mitigation(s):Use Azure Storage Analytics to audit access of Azure Storage. If possible, audit the calls to the Azure Storage instance at the source of the call. Refer: https://aka.ms/tmt-th20
SDL Phase:Implementation

8. An adversary can gain unauthorized access to resources in an Azure subscription  [State: Mitigation Implemented]  [Priority: High] 

Category:Elevation of Privileges
Description:An adversary can gain unauthorized access to resources in Azure subscription. The adversary can be either a disgruntled internal user, or someone who has stolen the credentials of an Azure subscription.
Justification:Our devices will be registered in the IoT Hub. A device cannot connect to the hub unless it has an entry in the identity registry. Symmetric key authentication will be used. Keys are only stored in the registry on the cloud, and a device will connect to IoT Hub through a connection string (access token) generated from its key. If required, a key can quicly be revoked for a device id by actioning an enable/disable flag in IoT Hub; in this manner, we can quickly disable access for an individual device if we suspect it has been tampered.
Possible Mitigation(s):Enable fine-grained access management to Azure Subscription using RBAC. Refer: https://aka.ms/tmtauthz#grained-rbac
SDL Phase:Design
